# 数据提取阶段方法剖析

> 记录时间：2026-04-12
>
> 本文档详细剖析 AutoSR 系统中数据提取环节的完整流程，从 PDF 文档解析到结构化数据输出。

---

## 整体架构概览

整个流程分为 **4 个阶段**，由 `ExtractionAgent`（`autosr/agents/extraction_agent.py`）统一编排：

```
PDF 文件
  │
  ▼
┌─────────────────────────┐
│  Step 1: Docling 文档解析  │  pdf_parser.py
└──────────┬──────────────┘
           ▼
┌─────────────────────────┐
│  Step 2: 分块 + BM25 检索  │  chunker.py
└──────────┬──────────────┘
           ▼
┌─────────────────────────────────────┐
│  Step 3: LLM 结构化提取 (GPT-5.4)    │  extraction_agent.py + prompts
│  ├─ 3A: 研究特征 (Characteristics)    │
│  └─ 3B: 研究结果 (Results)           │
└──────────┬──────────────────────────┘
           ▼
┌─────────────────────────┐
│  Step 4: 组装输出          │  ExtractionOutput
└─────────────────────────┘
```

---

## Step 1: Docling 文档解析

**对应文件**: `autosr/tools/pdf_parser.py`

### 数据流

```
PDF 文件 → Docling DocumentConverter → (Markdown 全文, 表格列表, 页数)
```

### 处理细节

- 使用 Docling 的 `DocumentConverter` 将 PDF 转换成 **Markdown 格式的纯文本**
- 开启 `do_table_structure=True` 和 `do_ocr=True`，确保表格结构被识别、扫描件也能处理
- 如果设置了环境变量 `DOCLING_ARTIFACTS_PATH`，则使用预下载的本地模型，无需联网
- **表格单独提取**：每个表格先尝试导出为 DataFrame → Markdown，失败则降级为 HTML
- 多个 PDF 通过 `parse_pdfs()` 批量处理，单篇失败不影响其他文件

### 输出数据结构 (`ParsedPDF`)

| 字段 | 类型 | 说明 |
|------|------|------|
| `filename` | str | 文件名 |
| `markdown_text` | str | 整篇论文的 Markdown 全文 |
| `tables` | List[str] | 表格列表（每个表格一个 Markdown 字符串） |
| `num_pages` | int | 页数 |

### 核心函数

- `parse_pdf(file_path)` → 解析单个 PDF，返回 `(markdown_text, tables, num_pages)`
- `parse_pdfs(file_paths)` → 批量解析，返回 `List[(filename, markdown_text, tables, num_pages)]`

## Step 2: 分块 + BM25 语义检索

**对应文件**: `autosr/tools/chunker.py`

### 目的

论文太长，不能全部塞给 LLM。需要筛选出与待提取字段最相关的片段，降低 token 开销和噪声。

### 2.1 分块 (`chunk_document`)

```
Markdown 全文 → 正文分块 (body_chunks)
表格列表     → 表格分块 (table_chunks)  每个表格独立成块
```

- 正文按 **1000 字符** 切分，**200 字符重叠**，在句号/问号等句子边界处断句
- 断句策略：在 chunk 末尾 20% 范围内寻找最近的句子结束符
- 每个表格单独作为一个 chunk，不做切分

### 2.2 BM25 检索 (`build_context_chunks`)

```
所有字段名+描述 → 拼成查询 → BM25 对 body_chunks 打分 → 取 top-k
                                                      + 全部表格
                                                      → 去重
```

- 把用户定义的所有字段名和描述拼成一个 BM25 查询字符串
- 使用 `rank_bm25` 库对正文块做相关性排序，取 **top-k**（默认 15）
- **表格全部保留**（因为数值型结果主要在表格里）
- 按文本内容 hash 去重，避免重叠区域导致的重复

### 2.3 格式化为 XML 引用格式 (`format_chunks_with_citations`)

最终输出格式：

```xml
<source id="1"><content>...第一段相关内容...</content></source>
<source id="2"><content>...第二段相关内容...</content></source>
```

这个格式方便 LLM 在提取时引用出处（citation），每个 source 有唯一编号。

### 重要说明

对每篇论文，分别为 **特征字段** 和 **结果字段** 各生成一份上下文（因为两类字段的检索关键词不同）：

- `char_contexts` — 用特征字段名检索得到的上下文
- `result_contexts` — 用结果字段名检索得到的上下文

### 核心函数

| 函数 | 说明 |
|------|------|
| `chunk_text(text, chunk_size, overlap, source)` | 将文本切分为重叠块 |
| `chunk_document(markdown_text, tables)` | 将文档分为正文块和表格块 |
| `retrieve_relevant_chunks(body_chunks, query, top_k)` | BM25 检索最相关的 top-k 块 |
| `build_context_chunks(body_chunks, table_chunks, field_names_and_descs, top_k)` | 构建最终上下文（BM25 + 全部表格 + 去重） |
| `format_chunks_with_citations(chunks)` | 格式化为 XML 引用格式 |

## Step 3: LLM 结构化提取

**对应文件**:
- `autosr/agents/extraction_agent.py` — 编排逻辑
- `autosr/prompts/extraction.py` — Prompt 模板
- `autosr/tools/llm.py` — LLM 调用层

### 核心机制

使用 **OpenAI Function Calling**（工具调用）让 LLM 以严格 JSON 格式返回提取结果，避免自由文本解析出错。模型默认为 `openai/gpt-5.4`（通过 OpenRouter 调用）。

### 3A: 提取研究特征 (Characteristics)

```
Prompt = PICO 背景 + 字段定义 + 论文片段
       ↓
GPT-5.4 (function calling: submit_characteristics)
       ↓
每篇论文 → 1 行，每个字段一个 FieldExtraction
```

- **每篇论文对应 1 行**
- Prompt 内容包含：
  - PICO 定义（P 人群、I 干预、C 对照、O 结局）
  - 字段列表（名称 + 描述）
  - BM25 筛选后的论文片段（XML 引用格式）
- LLM 通过 function calling 返回结构化数据，tool schema 强制要求每个字段都有值
- 提取不到的字段设为：`value="NOT FOUND"`, `confidence="LOW"`
- Prompt 模板变量：`{P}`, `{I}`, `{C}`, `{O}`, `{fields_text}`, `{chunks_text}`

### 3B: 提取研究结果 (Results)

```
Prompt = PICO 背景 + 字段定义 + 论文片段
       ↓
GPT-5.4 (function calling: submit_results)
       ↓
每篇论文 → N 行（按结局/亚组/时间点拆分）
```

- **每篇论文可以对应多行**（不同结局指标、亚组、时间点各一行）
- 每行有一个 `outcome_label`（如 `"Primary: BMI at 6 months"`）
- 特别强调扫描表格内容以获取数值结果
- 如果完全找不到结果，返回单行全 "NOT FOUND"

### 并行处理机制

- 3A 和 3B 分别用 `batch_function_call_llm()` 批量并行调用
- 通过 asyncio semaphore 控制并发数（默认 10），防止打爆 API
- 使用 tenacity 做指数退避重试（最多 5 次）
- 如果 LLM 未返回 tool_calls，尝试从 content 中解析 JSON 作为兜底

### 每个字段的提取结果 (`FieldExtraction`)

| 字段 | 类型 | 说明 |
|------|------|------|
| `field_name` | str | 字段名（如 "Sample Size"） |
| `value` | str | 提取值（如 "256"）或 "NOT FOUND" |
| `citation` | str | 原文引用（≤100 词的原文摘录） |
| `confidence` | str | HIGH / MEDIUM / LOW |

## Step 4: 组装输出

### 最终输出结构 (`ExtractionOutput`)

```json
{
  "characteristics": [
    {
      "filename": "PMID_12345678.pdf",
      "extractions": [
        {
          "field_name": "Author",
          "value": "Smith et al.",
          "citation": "Smith et al. conducted a randomized...",
          "confidence": "HIGH"
        },
        {
          "field_name": "Sample Size",
          "value": "256",
          "citation": "A total of 256 participants were enrolled...",
          "confidence": "HIGH"
        }
      ]
    }
  ],
  "results": [
    {
      "filename": "PMID_12345678.pdf",
      "outcome_label": "Primary: BMI at 6 months",
      "extractions": [
        {
          "field_name": "Effect Size",
          "value": "0.45",
          "citation": "The mean difference was 0.45...",
          "confidence": "HIGH"
        },
        {
          "field_name": "95% CI",
          "value": "[0.12, 0.78]",
          "citation": "95% confidence interval 0.12 to 0.78",
          "confidence": "MEDIUM"
        }
      ]
    }
  ]
}
```

### 输出数据模型

| 模型 | 说明 |
|------|------|
| `CharacteristicsRow` | 一篇论文一行：filename + extractions[] |
| `ResultsRow` | 一篇论文可多行：filename + outcome_label + extractions[] |
| `ExtractionOutput` | 最终输出：characteristics[] + results[] |
| `ExtractionSummary` | 进度统计：total_papers, papers_parsed, papers_extracted 等 |

## 系统输入

用户通过 API（`POST /api/extract` 或 `POST /api/extract/stream`）提交：

| 输入 | 类型 | 说明 |
|------|------|------|
| PDF 文件 | UploadFile[] | 需要提取的论文 |
| PICO 定义 | PICODefinition | P（人群）、I（干预）、C（对照）、O（结局） |
| 特征字段列表 | ExtractionFieldDefinition[] | 需要提取的研究特征（如 Author, Year, Sample Size...） |
| 结果字段列表 | ExtractionFieldDefinition[] | 需要提取的数值结果（如 Effect Size, 95% CI, P-value...） |
| top_k | int (1-30, 默认 15) | BM25 检索深度 |
| max_concurrency | int (1-50, 默认 10) | LLM 并发数 |

## 关键设计决策

| 决策 | 原因 |
|------|------|
| **BM25 做上下文门控** | 不把整篇论文丢给 LLM，先检索最相关的片段，降低 token 开销和噪声 |
| **表格永远全量保留** | 数值结果主要在表格中，不能被 BM25 过滤掉 |
| **Function Calling 强制结构化** | 用 tool schema 约束输出格式，不做自由文本解析，减少格式错误 |
| **特征和结果分开提取** | 两类数据的检索关键词和输出结构不同（特征一对一，结果一对多） |
| **Citation 追溯** | 每个提取值附带原文引用（≤100 词），支持人工审核验证 |
| **Semaphore 并发控制** | 防止并行请求过多打爆 OpenRouter API |
| **优雅降级** | PDF 解析失败、字段提取不到 → "NOT FOUND" + "LOW" confidence，流程继续不中断 |

## 涉及文件一览

| 层级 | 文件路径 | 职责 |
|------|---------|------|
| 文档解析 | `autosr/tools/pdf_parser.py` | PDF → Markdown + 表格 |
| 文本分块 | `autosr/tools/chunker.py` | 分块 + BM25 检索 + 格式化 |
| LLM 调用 | `autosr/tools/llm.py` | OpenRouter API 调用、并发控制、重试 |
| 提取编排 | `autosr/agents/extraction_agent.py` | 4 步流程编排 |
| Prompt 模板 | `autosr/prompts/extraction.py` | 特征提取和结果提取的 Prompt |
| 数据模型 | `autosr/schemas/extraction_models.py` | ParsedPDF、FieldExtraction、ExtractionOutput 等 |
| 共享模型 | `autosr/schemas/models.py` | PICODefinition 等 |
| 基础 Agent | `autosr/agents/base_agent.py` | 状态追踪、步骤记录 |
| HTTP API | `app/routers/extraction.py` | REST 接口（普通 + SSE 流式） |
| 配置 | `configs/settings.py` | API Key、模型名、Base URL |

## 流式输出 (SSE) 事件类型

通过 `POST /api/extract/stream` 接口可获得实时进度，事件类型如下：

| 事件类型 | 说明 |
|---------|------|
| `parsing` | 单篇 PDF 解析完成（ok/failed） |
| `parsing_done` | 全部解析完成（总数、成功数） |
| `chunking_done` | 分块完成（文档总数） |
| `extraction_start` | 开始提取（kind: characteristics/results） |
| `paper_extracted` | 单篇论文提取完成（含特征和结果） |
| `summary` | 提取统计摘要 |
| `done` | 全部完成，包含完整 ExtractionOutput |
| `error` | 异常信息 |
